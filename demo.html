01 · THE PROBLEM

LLMs hallucinate. Forensics can't afford it.

Most AI forensics tools wrap a tool like Volatility around an LLM and trust whatever comes out.

But in a real investigation, a confident wrong answer is worse than no answer. It can't survive scrutiny, and it can derail a case.

The question I built around:
What if the system didn't trust its own LLM?

02 · ARCHITECTURE

A dual-agent pipeline

INPUTWindows memory image (.raw)
↓
EVIDENCE COLLECTIONVolatility 3 — pslist, malfind, netscan, cmdline, dlllist
Every execution stored with a unique evidence ID + command + timestamp
↓
INVESTIGATOR AGENTReasons over evidence, chooses tools autonomously, cites evidence IDs on every finding
↓
VERIFIER AGENTIndependently checks each finding against cited evidence — accept or reject
↓
OUTPUTHTML report — accepted + rejected findings, fully traceable
03 · THE KEY IDEA

The verifier is independent

The verifier agent never sees the investigator's reasoning. It receives only two things:

  • The claim being made
  • The raw evidence that claim cited

It has no incentive to confirm the investigator. Like peer review, the reviewer doesn't know what answer the author wanted, only whether the evidence supports it.

This is what makes the self-correction real, not a scripted retry with a staged error.

04 · EVIDENCE TRACEABILITY

Every finding traces to a tool execution

Each finding cites evidence IDs. Each ID maps to an exact entry in the evidence store:

evidence_id: "EV-94b2231c"
command: "vol -r json -f image.raw windows.pslist"
timestamp: "2026-06-16T01:27:33Z"
parsed: [ ... ]

A judge can take any finding, find the cited ID in store.json, and read the exact command and output that produced it. Nothing is unverifiable.

05 · LIVE RESULT

The verifier caught all three

3

Example rejection, verbatim from the run:

REJECTED — "Claim states a single PID appears in both malfind and netscan, but the evidence shows multiple different PIDs that don't correlate."

The investigator overclaimed. The verifier checked it against the raw data and refused to sign off. That's the system working.

06 · HONEST LIMITATIONS

What's working, what isn't

Working

  • Full autonomous pipeline end to end
  • Evidence ID traceability on every claim
  • Independent verifier with substantive reasoning
  • Reproducible via disk + LLM caching

Not yet

  • Running 8B model on free budget, makes weak findings
  • Investigator loops on repeat filters
  • No re-investigation after rejection
  • Direct tool calls, not MCP transport

The verifier architecture is the contribution, and it holds regardless of which model sits behind it.

07 · WHAT'S NEXT

Where this goes

  • Ground investigator heuristics in actual evidence rows, not heuristic names
  • Feedback loop, rejected findings get re-investigated instead of dropped
  • Stronger model for the investigator, the verifier already discriminates well
  • MCP transport for standard SIFT tool integration

The core idea works today:
AI forensics that validates itself and won't present what it can't prove.
